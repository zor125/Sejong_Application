# Codex Rules

## 목적

이 문서는 Codex가 이 프로젝트에서 작업할 때 지켜야 할 규칙을 정의한다. Codex는 요청받은 범위를 우선하며, 불필요한 코드 변경이나 구조 변경을 하지 않는다.

## 기본 작업 규칙

- 한 번에 한 기능만 구현한다.
- 수정 가능한 폴더와 파일 범위를 반드시 지킨다.
- 기존 기능을 삭제하지 않는다.
- UI 스타일을 임의로 변경하지 않는다.
- 구현은 `docs/API.md`를 기준으로 한다.
- TypeScript `strict` 설정을 유지한다.
- `any` 사용을 최소화한다.
- mock data와 실제 API를 혼용하지 않는다.
- 작업 후 변경 파일 목록과 변경 이유를 요약한다.

## 영역별 수정 금지 규칙

- `admin-web` 작업 시 `student-app`, `backend`를 수정하지 않는다.
- `student-app` 작업 시 `admin-web`, `backend`를 수정하지 않는다.
- `backend` 작업 시 `admin-web`, `student-app`을 수정하지 않는다.
- 문서 작업 요청 시 코드 기능을 구현하지 않는다.
- 특정 파일만 수정하라는 요청이 있으면 해당 파일 외에는 수정하지 않는다.

## 문서 기준

- 제품 요구사항은 `docs/PRD.md`를 기준으로 한다.
- 화면 구조는 `docs/UI.md`를 기준으로 한다.
- DB 구조는 `docs/ERD.md`를 기준으로 한다.
- API 구조는 `docs/API.md`를 기준으로 한다.
- 작업 계획은 `docs/WBS.md`를 기준으로 한다.
- 문서와 구현이 충돌하면 문서를 먼저 업데이트하고 합의 후 구현한다.

## UI 작업 규칙

- 기존 레이아웃과 스타일 톤을 유지한다.
- 요청받지 않은 페이지의 UI를 변경하지 않는다.
- 화면에 필요한 mock data와 실제 API 호출을 동시에 사용하지 않는다.
- API 연결 전 단계에서는 mock data 사용 여부를 명확히 유지한다.
- 학생 앱 UI와 강사용 웹 UI를 혼동하지 않는다.

## API 작업 규칙

- 엔드포인트, Request, Response는 `docs/API.md`를 따른다.
- API 응답 필드명을 임의로 변경하지 않는다.
- 필요 없는 임시 필드를 추가하지 않는다.
- 웹과 앱에서 같은 개념은 같은 API와 같은 타입을 사용한다.

## 데이터 작업 규칙

- ERD의 용어와 관계를 따른다.
- ID는 UUID string으로 다룬다.
- `created_at`, `updated_at`, `deleted_at` 정책을 유지한다.
- soft delete 정책이 있는 데이터를 화면에서 기본 제외한다.

## 완료 보고 규칙

작업 완료 후 다음 내용을 요약한다.

- 생성 또는 수정한 파일 목록
- 각 파일의 변경 이유
- 실행 또는 검증한 명령
- 검증하지 못한 항목이 있으면 그 이유
